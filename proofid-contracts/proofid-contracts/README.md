# ProofID Contracts — `ProofIDRegistry`

A minimal, gas-conscious on-chain registry that anchors each student's
ProofID identity to their wallet, for the Polygon Amoy testnet. This is the
**blockchain layer only** — it is not yet wired into the ProofID frontend.

---

## Architecture

`ProofIDRegistry` is a single-contract registry built on OpenZeppelin's
`Ownable`. It intentionally stores as little as possible on-chain:

- **One profile per wallet.** `msg.sender` is always the profile owner —
  there's no separate "owner" parameter to spoof.
- **No large text or files on-chain.** The full ProofID profile (bio,
  projects, achievements, socials, avatar, etc.) lives off-chain. Only a
  `bytes32 profileHash` — the SHA-256 hash of that off-chain profile — is
  stored, so the chain can attest "this profile hasn't been tampered with"
  without paying to store it.
- **Two trust roles:**
  - The **wallet owner** can create their own profile once, and later
    update `profileHash`, `department`, and `graduationYear`.
  - The **contract owner** (deployer, or whoever they transfer ownership
    to — e.g. a university admin key or ProofID's verification service)
    can mark a student `verified` or revoke that verification. The
    contract owner cannot create, edit, or delete a student's profile.

### Why `fullName` and `university` are immutable after creation

They're treated as the identity anchor for the profile. Letting a wallet
silently rewrite "who I am" after being verified would undermine the point
of verification. If a student's name or university genuinely changes, that
should be a deliberate re-verification flow in a later phase, not a
same-function update.

## Storage layout

```solidity
struct Student {
    address walletAddress;   // 20 bytes ┐
    uint16  graduationYear;  //  2 bytes │ packed into a single slot
    bool    isVerified;      //  1 byte  │ (24 of 32 bytes used)
    bool    exists;          //  1 byte  ┘
    string  fullName;        // dynamic, own slot(s)
    string  university;      // dynamic, own slot(s)
    string  department;      // dynamic, own slot(s)
    bytes32 profileHash;     // 32 bytes, own slot
    uint40  createdAt;       // 5 bytes ┐ packed into a single slot
    uint40  updatedAt;       // 5 bytes ┘
}
```

`walletAddress`, `graduationYear`, `isVerified` and `exists` pack into one
32-byte slot; `createdAt`/`updatedAt` pack into another. `uint40` comfortably
holds Unix timestamps until the year 36812, so there's no realistic overflow
risk while still saving a slot versus `uint256`.

Custom errors (`ProfileAlreadyExists`, `EmptyName`, `InvalidGraduationYear`,
etc.) are used throughout instead of `require(string)` to save deployment
and revert gas.

## Functions

| Function | Caller | Effect |
|---|---|---|
| `createProfile(fullName, university, department, graduationYear, profileHash)` | anyone, once | Creates `msg.sender`'s profile. Reverts if one already exists or inputs are invalid. |
| `updateProfile(newProfileHash, newDepartment, newGraduationYear)` | profile owner | Updates hash/department/year and `updatedAt`. |
| `verifyStudent(wallet)` | contract owner | Sets `isVerified = true`. |
| `removeVerification(wallet)` | contract owner | Sets `isVerified = false`. |
| `getProfile(wallet)` | anyone (view) | Returns the full profile tuple. |
| `profileExists(wallet)` | anyone (view) | Returns `bool`. |
| `getProfileHash(wallet)` | anyone (view) | Returns just the `bytes32` hash. |

### Events

`ProfileCreated`, `ProfileUpdated`, `StudentVerified`, `VerificationRemoved` —
each indexed by wallet address so off-chain indexers can filter cheaply.

## Project structure

```
contracts/          ProofIDRegistry.sol
scripts/deploy.ts    plain ethers deployment script
ignition/modules/     Hardhat Ignition deployment module (alternative to scripts/deploy.ts)
test/                 Hardhat + chai test suite
hardhat.config.ts     network, solidity, etherscan/verify config
.env.example          required environment variables
```

`artifacts/`, `cache/`, and `typechain/` are build outputs and are
regenerated by `npm run compile` — they are not committed.

## Setup

```bash
npm install
cp .env.example .env
# fill in PRIVATE_KEY, RPC_URL, POLYGONSCAN_API_KEY in .env
```

Get testnet POL for your deployer wallet from the
[Polygon Amoy faucet](https://faucet.polygon.technology/) before deploying.

## Compiling

```bash
npm run compile
```

## Testing

```bash
npm test
```

The suite covers profile creation (including every validation rule),
duplicate-profile rejection, unauthorized updates, owner-only verification
and verification removal, hash storage/retrieval, profile retrieval for
both the owner and third parties, and multi-profile edge cases. All tests
are expected to pass (38/38 at time of writing).

## Deploying to Polygon Amoy

Two equivalent options are provided:

```bash
# Option A — plain ethers deploy script (prints address, network, tx hash)
npm run deploy:amoy

# Option B — Hardhat Ignition (declarative, resumable deployments)
npx hardhat ignition deploy ignition/modules/ProofIDRegistry.ts --network polygonAmoy
```

Network details (already configured in `hardhat.config.ts`):

| | |
|---|---|
| Chain name | Polygon Amoy |
| Chain ID | `80002` |
| Currency | POL |
| RPC URL | `https://rpc-amoy.polygon.technology` |
| Block explorer | `https://amoy.polygonscan.com` |

## Verifying on Polygonscan

```bash
npx hardhat verify --network polygonAmoy <CONTRACT_ADDRESS> <INITIAL_OWNER_ADDRESS>
```

(The deploy script prints this exact command with the addresses filled in
once deployment finishes.) This requires `POLYGONSCAN_API_KEY` in `.env` —
a standard Polygonscan API key works across Polygon networks including Amoy.

## Explicitly out of scope for this phase

No smart-contract writes are wired to the frontend yet, and this contract
does not implement: profile storage beyond the hash, IPFS, a database,
a backend/auth server, message signing, or NFTs. Those are later phases.
